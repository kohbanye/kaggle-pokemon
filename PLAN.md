# PLAN — Pokémon TCG AI Battle 攻略プラン

調査(`docs/research/game-ai-survey.md`)を踏まえた段階プラン。
方針: **探索系（決定化/ISMCTS）を軸に、価値関数を学習で差す AlphaZe\*\*型ハイブリッド**へ向け、
**ablation（1要素ずつ足し引きして効果を測り、使う/使わないを切り分ける）** を徹底する。

---

## 基本原則（全Phase共通）

### A. ablation の規律
- **一度に変える変数は1つ**。エージェント比較時はデッキ固定、デッキ比較時はエージェント固定。
- すべての手法は **「前段ベースラインに対する勝率」** で評価し、**有意差が出たものだけ採用**する。
- 各実験は **keep / drop / 保留** を必ず判定し、§ 採否台帳 に記録する（なぜ採用/不採用かも）。
- 「効きそう」「論文で強い」は理由にしない。**この環境で測って勝ったものだけ残す**。

### B. 評価方法論（measurement）
- 比較は **同一シード・同一デッキ・先後を入れ替えて各 N 戦**（初期値 **N≥400**、僅差判定は 1000+）。
- 指標は **勝率 + Wilson 95%信頼区間**。CI が 50% を跨がない＝有意。僅差は試合数を増やす。
- **分散低減**: 可能な限り同一決定化シード/同一対戦カードで paired 比較。
- 採用の足切り（初期値、運用しながら調整）:
  - **明確採用**: 勝率 ≥ 55%（CIが50%超）
  - **保留/再計測**: 51–55%（試合数増やして判断、計算コストと相談）
  - **不採用**: ≤ 51% もしくは CI が 50% を含む

### C. ローカル sim と実ラダーの二層評価
- ローカル sim は **高速な相対比較**用。ただし **実ラダー結果を外す**ことが知られる → 最終判断は実ラダー。
- **大きな意思決定（デッキ確定・探索採否・最終提出）は必ずラダーで検証**（5サブ/日の枠を割く）。
- ローカル順位とラダー順位の **ズレを記録**し、ローカル評価のバイアスを把握する。

### D. 提出衛生（毎Phase維持）
- **絶対にクラッシュしない**（例外時は合法手フォールバック）／**手番時間 ＆ 10分/試合を超えない**。
- 提出は `main.py`(トップ階層) + `deck.csv` + `cg/` の自己完結バンドル（ネット無し・CPU前提）。
- どのPhaseの版でも「いつでも提出できる動く状態」を保つ（退行したらすぐ気づける）。

## Phase 0 — 評価基盤とベースライン
**目的**: 以降すべての ablation を同じ土俵で測れる、再現可能な対戦評価環境を作る。

**作るもの**
- Docker(linux/amd64) 上の **対戦ランナー**: 2エージェント×Nゲーム、先後入替、シード固定、勝率+CI出力。
- **エージェント登録**の仕組み（`agents/` に差し替え可能な方策を置く）。
- ベースライン2種: `random`（合法手ランダム）/ `greedy`（1手の即時報酬最大、例: 取れるKOを取る）。
- 結果ログ（CSV/JSON）と簡易集計（勝率・CI・平均ターン・時間/手）。

**達成基準（Exit）**
- random vs random ≈ 50%（±CI内）＝ハーネスが公平。
- `greedy` が `random` を **有意に**上回る（勝率≥55%, 500戦）。
- 同一シードで結果が再現する。1手あたり・1試合あたりの所要時間を計測できている。

**Ablation**: ここは“ものさし”作り。**有意差の閾値・必要試合数を確定**させる（B項を実データで較正）。

## Phase 1 — デッキ選定（最大のレバー）
**目的**: 「デッキ選択がエージェントの質を支配する」前提に従い、**強いベースラインデッキを1〜3個確定**する。

**作るもの**
- 候補デッキ群（サンプルデッキ＋メタ/コミュニティ由来＋EDA知見から数〜十数個）。
- **エージェント固定**（Phase0 `greedy` か簡易探索）でデッキ総当たり（round-robin）。
- 各デッキの相性表（誰に強い/弱い）。

**達成基準（Exit）**
- **主力デッキ1個＋対抗1〜2個**を選定し、選定理由（相性・安定性）を文書化。
- 「デッキ差」が「ここまでのエージェント差」より勝率を動かすことを定量確認（レバーの大きさの裏取り）。
- **初回ラダー提出**を行い、ローカル順位とラダーの対応を取り始める（二層評価の起点）。

**Ablation**: エージェント固定でデッキだけ変える。**勝てないデッキを切る**。
相性が割れる場合は「主力＋メタ対抗」を残し、单一デッキへの過剰最適化を避ける。

---

## Phase 2 — ヒューリスティック評価関数（探索なし）
**目的**: 盤面/行動を点数化する評価関数を作り、**探索なしで** greedy を明確に上回る。探索の“土台”を作る。

**作るもの**
- 状態評価の特徴量: prize差・場の総HP/期待打点・エネ付与状況・KO可否/被KOリスク・ベンチ展開・手札リソース等。
- 行動スコアリング（1手 or 浅い貪欲）。各特徴を **on/off 切り替え可能**に実装（ablation前提の設計）。

**達成基準（Exit）**
- `heuristic` が Phase0 `greedy` を有意に上回る（≥55%）。
- **特徴量ごとの寄与**が分かっている（どれが効き、どれが無駄か）。

**Ablation（特徴量レベル）**: 各特徴を1つずつ抜いて勝率変化を測り、**効く特徴だけ残す**。
無寄与・有害な特徴は削除（評価関数を軽く・速く保つ＝探索予算の節約）。

---

## Phase 3 — 決定化 + 木探索（PIMC）
**目的**: `search_begin/search_step` で **決定化＋深さ制限探索**を入れ、**ヒューリスティック単体を上回るか**を検証。

**作るもの**
- PIMC: 隠れ情報(相手手札/山札/サイド)を予測サンプル→完全情報として短い探索→平均。
- リーフ評価に Phase2 の評価関数を使用。**時間予算管理**（手番上限・10分/試合厳守、途中打ち切り）。

**達成基準（Exit）**
- `PIMC` が Phase2 `heuristic` を有意に上回る（≥55%）**かつ** 時間予算内。
- **決定化数 × 探索深さ** のスイープで「効果が頭打ちになる点(knee)」を把握＝コスト対効果が分かる。

**Ablation**: 探索 ON/OFF、決定化数{1,5,20,…}、深さ{1,2,3,…}を振る。
**KEEP条件: ローカル＆ラダー両方でヒューリスティックを上回る**こと。
⚠️ 調査の警告: 分岐因子が大きいと探索が決定化UCTと **互角**になりうる → 「探索は当然強い」と仮定せず**実測で採否**。
時間内に有意な利得が出なければ **drop**（評価関数強化に投資を戻す）。

---

## Phase 4 — ISMCTS（戦略融合の緩和）
**目的**: PIMC を **ISMCTS**（情報集合の単一木）に置換し、**戦略融合の緩和が実利益になるか**を検証。

**作るもの**
- 情報集合をノードとする MCTS。**同一計算予算**で PIMC と差し替え可能に。

**達成基準（Exit）**
- `ISMCTS` vs `PIMC` を **等計算予算**で頭対頭比較し、採否を確定。

**Ablation（核心の切り分け）**: ISMCTS vs PIMC、等予算で勝負。
- 有意に勝つ → **採用**。
- 互角/僅差（調査が高分岐ゲームで警告する結果）→ **PIMCに戻す**（実装/保守コストの軽い方を残す）。
これは「使う方が良い手法/使わない方が良い手法」を最も綺麗に切り分けるPhase。**結論をはっきり記録**。

---

## Phase 5 — ニューラル価値/方策（AlphaZe\*\*型）※Phase3/4が頭打ちなら
**目的**: オフライン学習した **価値関数（と方策prior）** を探索に差し、リーフ評価/手順序付けを強化する。

**前提/ゲート**: Phase3-4 の探索が有効、かつ伸びが頭打ち。オフライン自己対戦データ生成基盤（Docker sim、可能ならGPU）が用意できること。整わなければ **このPhaseは見送り**、Phase2/3の磨き込みに戻す。

**作るもの**
- 自己対戦データ生成 → **価値ネット**（＋任意で方策ネット）学習。
- 探索への統合: リーフ評価をヒューリスティック→学習価値に、手順序付けに方策prior。
- 状態表現の設計（カード埋め込み／場・手札・サイドの集合エンコード）。

**達成基準（Exit）**
- `learned-value + search` が `heuristic + search` を有意に上回る、**かつ CPU推論が時間内**。

**Ablation**: 同一探索内で **評価=ヒューリスティック vs 学習価値**、**方策prior on/off** を比較。
**効いた構成要素だけ採用**。学習価値が時間/精度で割に合わなければ heuristic を維持。

---

## Phase 6 — 軽量化・頑健化・最終提出
**目的**: 制限時間に余裕を持って収め、**過剰適合を避けた頑健な最終版**を確定する。

**作るもの**
- 必要なら **方策蒸留**: 重い探索/モデルの出力を軽量方策に模倣学習で写し、推論を高速化。
- 時間予算ハードニング（最悪ケースでも超過しない）、フォールバック網羅、リーグ的多相手評価。

**達成基準（Exit）**
- 手番時間/10分に**マージンを持って**収まる・**全局面でクラッシュしない**。
- **多様な相手**（自作の各版＋メタ対抗デッキ）に対して頑健、ラダーでも退行なし → 最終提出。

**Ablation**: **蒸留版 vs フル探索版**（蒸留で失う強さは許容内か）／**多相手ロバスト性**（単一相性への過剰適合チェック）。

## 手法の採否仮説（事前の見立て・最終は実測で更新）

| 手法 | 事前見立て | Phaseで判定 |
|---|---|---|
| 強いデッキ | **使う**（最大レバー） | P1 |
| ルールベース/合法手フォールバック | **使う**（必須・安全網） | P0, 全Phase |
| ヒューリスティック評価関数 | **使う**（探索の土台） | P2 |
| 決定化 + 木探索(PIMC) | **使う寄り**（探索APIに直結） | P3 |
| ISMCTS | **計測して判断**（高分岐だとPIMCと互角の警告） | P4 |
| ニューラル価値(AlphaZe\*\*型) | **計測して判断**（学習基盤が要る） | P5 |
| 方策prior / 手順序付け | **計測して判断** | P5 |
| 方策蒸留(distillation) | **計測して判断**（軽量化の手段） | P6 |
| フル MuZero 系を自前学習 | **使わない寄り**（GPU/学習基盤が重すぎ） | — |
| CFR/Deep CFR/ReBeL のフル均衡 | **使わない寄り**（実装/計算が重い。発想だけ借用） | — |
| DeepNash の R-NaD フル学習 | **使わない寄り**（学習大規模。推論軽い点は魅力だが学習コスト過大） | — |
| LLM を推論ループ内で使用 | **使わない**（CPU/時間/ネット制約で非現実的。設計役はオフラインで活用） | — |

> 「使わない寄り」でも **アイデアは借用**する: 確率性→*Stochastic MuZeroのafterstate*、付け込まれにくさ→*均衡(DeepNash/ReBeL)*。

### 較正メモ（Phase 0 で確定した運用値）

- **エンジン乱数は seed 不能**: 公開API（`battle_start`）にシード引数が無く、同一エージェントseedでも局ごとに展開が変わる（実測: 同seedで52手 vs 15手）。→ **局単位のbit再現は不可能**。エージェント側乱数のみseedし、全結果をCSVに記録、**先後入替＋大N＋Wilson CI** で統計的に再現する運用とする（採否判定はこれで足りる）。
- **必要試合数の目安**（Wilson CI 半幅から逆算）:
  - **N=500**: 半幅 ≈ ±4.4pp。**55%級の差はギリ有意**（55%なら下限≈50.6%）。明確採用/不採用の一次判定はこれで十分。
  - **僅差（52–53%級）**: N=500では50%を跨ぐ → **N≈1,000–2,400** が必要（§B「僅差は1000+」を定量裏付け）。
  - 運用: まずN=500でスクリーニング、跨いだら増やす。
- **計測コスト**: Docker(linux/amd64エミュ)で **約180–340 games/sec**（greedy同士は短手数で速い）。500戦≈1.5–2.8秒。ローカルでの大量ablationは安価。
- **手番時間**: 両ベースラインとも1手 **平均~0.002ms / 最悪~0.2ms**。制限時間に対し桁違いに余裕（探索を載せる予算が十分にある）。
- **足切り閾値は §B の初期値のまま据え置き**（明確採用≥55% / 保留51–55% / 不採用≤51%）。N=500で運用可能と確認できたため変更不要。

## マイルストーン（目安）
- **M1**: P0–P1 完了 → 評価基盤＋主力デッキ確定＋初ラダー提出（ローカル↔ラダー較正開始）
- **M2**: P2–P3 完了 → ヒューリスティック＋探索でラダー中位を狙える版
- **M3**: P4 完了 → ISMCTS採否を確定（綺麗なablation結論）
- **M4**: P5–P6 完了 → 学習価値ハイブリッド＋頑健化した最終提出（〆切 2026-08-16）
